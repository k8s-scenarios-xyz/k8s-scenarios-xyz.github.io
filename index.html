Scenario A: (1) Cluster Management K8s v1.2.4, multi-tenant namespace setup

Zombie Pods Causing NodeDrain to Hang

Cluster workloads failed after applying overly strict resource quotas that denied new pod creation.

Medium Severity K8s v1.23
What Happened

A new quota was applied with very low CPU/memory limits. All new pods across namespaces failed scheduling.

Root Cause

The resource quota was set too low for the cluster's minimum requirements, preventing any new pods from being scheduled.

Solution

Edited the ResourceQuota object to increase the limits:

kubectl edit quota my-quota -n my-namespace

Increased the CPU and memory limits to appropriate values for the cluster.

Prevention
  • Test resource quotas in a non-production environment first
  • Set realistic limits based on actual pod requirements
  • Monitor resource usage regularly

Scenario B: (2) Cluster Management K8s v1.24, GKE, custom controllers

API Server Crash Due to Excessive CRD Writes

API server became unresponsive due to excessive load from Custom Resource Definition operations.

High Severity K8s v1.24
What Happened

The API server began crashing repeatedly, making the cluster unavailable. This occurred after deploying a new custom controller.

Root Cause

A misconfigured custom controller was creating thousands of CRD updates per second, overwhelming the API server.

Solution

Identified and deleted the problematic controller:

kubectl get deployments -n kube-system
kubectl delete deployment faulty-controller -n kube-system

Restarted the API server to clear the pending requests.

Prevention
  • Implement rate limiting in custom controllers
  • Test controllers under load before deploying to production
  • Monitor API server metrics closely after controller deployments

Scenario C: (3) Cluster Management K8s v1.21, Self-managed, Static nodes

Node Not Rejoining After Reboot

After a planned reboot, a worker node failed to rejoin the Kubernetes cluster.

Medium Severity K8s v1.21
What Happened

After rebooting a worker node for maintenance, the node failed to rejoin the cluster. The kubelet was running but not registering with the control plane.

Root Cause

The kubelet had lost its credentials and could not authenticate with the API server. This was due to certificate expiration on the node.

Solution

Regenerated kubelet credentials on the affected node:

sudo kubeadm alpha certs renew all
sudo systemctl restart kubelet

Verified node rejoined the cluster with:

kubectl get nodes
Prevention
  • Regularly check certificate expiration in the cluster
  • Implement automated certificate rotation
  • Monitor node status after reboots
